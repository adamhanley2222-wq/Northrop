MD Strategic Review AI — Full Build-Spec Pack
1. Product summary
Build a web application that ingests annual strategic plans and quarterly business reports, preserves their organisational and strategic structure, and lets executive users ask grounded, strategy-aware questions across periods, divisions, leaders, and objectives.
The application is a working demo for a managing director and leadership team. It must not behave like a generic document chatbot. It must behave like a strategic review assistant that:
understands reporting hierarchy
compares quarter to quarter
maps delivery evidence to annual strategy
highlights gaps, tensions, and unresolved issues
proposes sharper executive follow-up questions
grounds all responses in source evidence
---
2. Primary demo objective
The first demo must prove that the system can:
ingest annual strategy and quarterly reports
preserve report structure and organisational context
answer questions across both document types
compare like-for-like themes quarter to quarter
trace performance and commentary back to annual strategic objectives
identify missing evidence, reporting gaps, or strategic drift
generate MD-grade challenge questions

5. Product principles
Source first  
Every answer must tie back to retrieved source evidence.
Structure over flat text  
Documents must be parsed according to headings, sections, tables, risks, actions, and commentary blocks.
Strategy-aware retrieval  
The system should retrieve based on period, business unit, leader, objective, and section type, not just embeddings.
Comparison is a first-class feature  
Q-on-Q comparison must be deliberate, not accidental.
Challenge, don’t just summarise  
Outputs should include gaps, tensions, and follow-up questions.
Human-correctable metadata  
Admin users must be able to fix ambiguous or incorrect tags.

11. Ingestion pipeline spec
Step 1: Upload
API endpoint accepts PDF or DOCX.
System actions:
save file
compute file hash
create document record
mark parse status as pending
Step 2: Parse document
For PDFs:
extract page text
detect headings where possible
extract tables
For DOCX:
preserve heading hierarchy
preserve paragraphs and tables
If extraction quality is low, flag document for review.
Step 3: Section segmentation
Segment into sections based on:
heading changes
subheading changes
tables with context text
bullets and numbered action blocks
Step 4: Chunking
Chunk using structure-aware logic:
do not split inside short coherent sections unnecessarily
group table commentary with the relevant table summary
keep risk lists and action lists as independent chunks where possible
Chunk size guidance
target 250 to 800 tokens
allow smaller chunks for concise risk/action items
avoid giant chunks over 1200 tokens except in fallback mode
Step 5: Chunk classification
Classify each chunk for:
chunk type
likely division/business unit
likely strategic pillar/objective
likely leader/owner
likely reporting period
signal type
extraction confidence
This step may use deterministic rules first, then LLM assistance.
Step 6: Canonical mapping
Generate canonical objects from chunks where appropriate.
Examples:
a section with narrative on project delivery becomes an Observation
a section indicating slippage becomes PerformanceSignal + possibly Risk
a stated next step becomes ActionItem
a statement linked to growth objective becomes EvidenceLink to that StrategicObjective
Step 7: Embedding
For every published chunk:
create embedding
upsert to Pinecone
include metadata fields for filterable dimensions
Step 8: Publish
Document becomes queryable only after successful parse and indexing, unless admin manually overrides.

13. Retrieval architecture
13.1 Query intent types
Supported intents:
summary
comparison
strategy_alignment
gap_detection
risk_review
executive_questions
cross_division_theme
13.2 Retrieval stages
Stage A: Intent classification
Classify question into one primary intent and optional secondary intent.
Stage B: Filter resolution
Infer or apply filters from the question:
quarter / year
division
leader
strategic pillar or objective
document type
Stage C: Hybrid candidate retrieval
Use:
metadata-filtered semantic search from Pinecone
optional direct SQL fetch of canonical objects in the same scope
Stage D: Intent-specific evidence assembly
Examples:
comparison intent should deliberately retrieve corresponding periods
gap_detection should retrieve strategy objects and check for weak/no reporting evidence
executive_questions should retrieve risks, weak evidence, and inconsistent signals
Stage E: Answer synthesis
Answer must include:
direct answer
supporting evidence summary
specific cited sources
gaps/tensions
suggested executive follow-up questions

15. Prompt modules
Use separate prompt modules rather than one monolithic prompt.
15.1 Ingestion classification prompt
Purpose:
classify chunk type
infer organisational and strategic tags
identify whether chunk maps to Observation, Risk, Opportunity, ActionItem, PerformanceSignal
Expected output JSON:
```json
{
  "chunk_type": "risk",
  "section_type": "risk_block",
  "division_name": "Structures",
  "business_unit_name": null,
  "leader_name": null,
  "strategic_pillar_name": "Operational Excellence",
  "strategic_objective_name": "Improve delivery predictability",
  "observation_type": null,
  "status": "amber",
  "trend": "worsening",
  "risk_severity": "high",
  "confidence": 0.78,
  "rationale": "..."
}
```
15.2 Strategic mapping prompt
Purpose:
map quarterly evidence to annual strategic objectives
identify aligned, weakly aligned, or unaligned content
Expected output JSON:
```json
{
  "links": [
    {
      "strategic_objective_name": "Improve delivery predictability",
      "link_type": "aligned_to",
      "confidence": 0.84
    }
  ],
  "unmapped_reason": null
}
```
15.3 Query intent prompt
Purpose:
classify the user question into an intent
extract filters
Expected output JSON:
```json
{
  "primary_intent": "comparison",
  "secondary_intent": "strategy_alignment",
  "filters": {
    "financial_year": 2026,
    "quarters": ["Q2", "Q3"],
    "division_name": "Structures",
    "leader_name": null,
    "strategic_pillar_name": null,
    "strategic_objective_name": null
  }
}
```
15.4 Answer synthesis prompt
Purpose:
synthesize an answer only from retrieved evidence
present facts, inferences, gaps, and executive questions clearly
Rules:
do not use facts outside provided evidence
explicitly say when evidence is incomplete
include source references using chunk/page labels supplied by backend
15.5 Executive challenge prompt
Purpose:
generate tough but constructive follow-up questions
focus on risk, drift, inconsistent reporting, missing evidence, and accountability
Expected output JSON:
```json
{
  "questions": [
    "What corrective action was taken after the delivery slippage first appeared in Q2?",
    "Why does the report reference utilisation pressure without quantifying the impact on margin or schedule?"
  ]
}
```

21. Processing rules
Rule 1
Never answer from vector snippets alone if full chunk or section context is needed.
Rule 2
When a query is comparative, explicitly gather data from each target period before synthesis.
Rule 3
When evidence is absent for a strategic objective, say so clearly.
Rule 4
Store confidence separately from content. Do not overwrite uncertain extraction with certainty.
Rule 5
Admin overrides take precedence over auto-inferred metadata.
Rule 6
Keep raw source text immutable once extracted. Store corrections as metadata overrides, not text edits.
